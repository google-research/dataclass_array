# Design: SSE for Todos — Agent Manager v2

*Date: 2026-04-02*

## Summary

Currently, the Todos plugin in the frontend polls the backend every 10 seconds
(the user reported 5 seconds, but code shows 10 seconds) to refresh the todo
list. This design proposes replacing polling with **Server-Sent Events (SSE)**
pushed by the sidecar server to the frontend. This reduces network traffic and
provides real-time updates when todos are created, updated, or deleted.

--------------------------------------------------------------------------------

## Proposed Changes

### 1. Backend (Sidecar - Go)

We will implement a simple PubSub/Hub mechanism in the `todos` plugin to manage
client connections and broadcast events.

#### File: `plugins/todos/sidecar/plugin.go`

-   **Define `Hub`**: A struct to manage active SSE client channels and
    broadcast messages.
-   **Add `/api/todos/events` endpoint**:
    -   Sets headers for SSE (`text/event-stream`, `no-cache`, `keep-alive`).
    -   Registers the client channel.
    -   Streams events.
    -   Handles disconnection via `r.Context().Done()`.
-   **Broadcast on Mutations**:
    -   `handleTodo` (POST): Broadcast `{"type": "todos_updated"}` or `{"type":
        "todo_created", "id": "..."}`.
    -   `handlePatchTodo` (PATCH): Broadcast `{"type": "todos_updated"}`.
    -   `handleDeleteTodo` (DELETE): Broadcast `{"type": "todos_updated"}`.

For simplicity and robustness, we can start by broadcasting a generic
`"todos_updated"` event, which triggers a refetch on the client. This is less
prone to state desynchronization than sending partial diffs.

#### Draft Implementation Sketch

```go
type Hub struct {
    clients    map[chan string]bool
    register   chan chan string
    unregister chan chan string
    broadcast  chan string
}

// ... NewHub and Run methods ...

type Plugin struct {
    r   *repo
    hub *Hub
}

func (p *Plugin) Init(ctx context.Context, pool *pgxpool.Pool) error {
    // ... existing init ...
    p.hub = NewHub()
    go p.hub.Run()
    return nil
}

func (p *Plugin) RegisterRoutes(mux *http.ServeMux) {
    // ... existing routes ...
    mux.HandleFunc("/api/todos/events", p.handleEvents)
}

func (p *Plugin) handleEvents(w http.ResponseWriter, r *http.Request) {
    w.Header().Set("Content-Type", "text/event-stream")
    w.Header().Set("Cache-Control", "no-cache")
    w.Header().Set("Connection", "keep-alive")

    clientChan := make(chan string)
    p.hub.register <- clientChan

    defer func() {
        p.hub.unregister <- clientChan
    }()

    flusher, ok := w.(http.Flusher)
    if !ok {
        http.Error(w, "Streaming unsupported!", http.StatusInternalServerError)
        return
    }

    fmt.Fprintf(w, "data: connected\n\n")
    flusher.Flush()

    for {
        select {
        case <-r.Context().Done():
            return
        case msg := <-clientChan:
            fmt.Fprintf(w, "data: %s\n\n", msg)
            flusher.Flush()
        }
    }
}
```

### 2. Frontend (React - TypeScript)

We will use the browser's native `EventSource` to listen for events from the
sidecar.

#### File: `plugins/todos/plugin.tsx`

-   Replace `setInterval` with `EventSource` connection.
-   On message, call `fetchTodos`.

#### Draft Implementation Sketch

```typescript
useEffect(() => {
  const ws = currentWs;
  const eventSource = new EventSource(`/${ws}/api/todos/events`);

  eventSource.onmessage = (event) => {
    if (event.data === 'todos_updated' || event.data === 'connected') {
      fetchTodos();
    }
  };

  eventSource.onerror = (err) => {
    console.error('SSE error:', err);
    eventSource.close();
    // Fallback or retry logic if needed
  };

  return () => {
    eventSource.close();
  };
}, [fetchTodos, currentWs]);
```

## Open Questions / Considerations

1.  **Authentication / CSRF**: Does `EventSource` need custom headers? By
    default, `EventSource` does not support custom headers (like
    `x-codeium-csrf-token`). However, cookies are sent automatically. If the
    sidecar relies on cookies for session/auth, it should work. If it requires
    the CSRF token header (which we saw in ConnectRPC), we might need to pass it
    as a query parameter or use a `fetch` + `ReadableStream` approach instead of
    `EventSource`.
    -   *Investigation needed*: Check if sidecar routes require
        `x-codeium-csrf-token` or if that is only for the Language Server. The
        sidecar routes seem to be plain HTTP and might rely on the Gateway for
        protection or just use cookies.
2.  **Granularity of Events**: Should we send the updated Todo object in the SSE
    stream to avoid a refetch?
    -   *Recommendation*: Start with a simple "refresh" signal to ensure
        correctness. Optimizing to send full objects can be done later if
        network overhead of refetching is an issue (unlikely for a todo list).
